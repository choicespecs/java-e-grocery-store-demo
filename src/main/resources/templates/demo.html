The Checkout Saga — 6 Steps
Step 1 — checkAvailability Inventory read-only
Pre-flight check — no stock is held yet, nothing is charged. If DOWN → immediate fail, zero side effects. Safe to retry.
Step 2 — applyPromotion Promotion read-only non-critical
Validates a promo code and calculates a discount. Skipped if no code entered.
Graceful degrade ON: if DOWN → step logged as DEGRADED, checkout continues at full price.
Graceful degrade OFF: if DOWN → checkout aborts, no side effects.
Step 3 — reserveStock Inventory side effect
Stock is now held for this order. Other sessions cannot buy these units. If DOWN → fail before payment — no compensation needed.
Step 4 — processPayment Payment side effect dedicated thread + timeout
Card is charged on a separate thread with a hard deadline. If DOWN, declined, or timed out → COMPENSATION: releaseReservation — stock returned to pool, no money moved.
Step 5 — createOrder Database side effect
Order persisted to H2. If DB write fails → COMPENSATION: refund + releaseReservation.
Step 6 — commitReservation Inventory side effect
The reservation becomes permanent (hold record removed). Hardest failure point — both payment and order already exist. If DOWN → COMPENSATION: refund + cancelOrder + releaseReservation.

Design Principles Illustrated
1 — Saga Pattern (Compensating Transactions)

A distributed checkout cannot be wrapped in a single ACID transaction — payment and inventory live in separate systems. The Saga pattern sequences local operations and defines an explicit compensating action for each step that has side effects, so any failure can be undone in reverse order.

Caveat: compensations are themselves fallible. Production systems use an outbox table — compensating events are persisted first, then applied by a background worker with retry + exponential back-off, so even a crashed JVM cannot lose them.

2 — Graceful Degradation (Critical vs. Non-Critical Dependencies)

The key question: "Can the core business action succeed without this service?" Payment and inventory are critical — their failure must abort checkout. The promotion service is non-critical — a customer can still buy without a discount. When it is DOWN, the saga logs a DEGRADED step and continues rather than rolling back.

In production: always emit a metric or alert when you degrade, even though the customer journey succeeds. Silent degradation hides outages from on-call engineers.

3 — Timeouts, Threads, and Circuit Breakers

Payment runs on a dedicated daemon thread with a hard deadline (3000 ms). If it exceeds that limit, the future is cancelled and releaseReservation fires. This prevents a slow payment gateway from holding a servlet thread indefinitely.

Inventory has no such protection — it runs directly on the request thread. Try the Inventory SLOW at reserve preset: the entire request stalls for 4 seconds. Under heavy load, a slow inventory service would exhaust the servlet thread pool and bring down the whole application.

In production: wrap every external call in a timeout, not just payment. Add a Resilience4j @CircuitBreaker: after N consecutive timeouts the breaker opens and subsequent calls fail fast (no thread blocking) until the service recovers.

4 — Idempotency Keys and Retry Safety

A UUID is embedded as a hidden field when the checkout form loads. The payment service caches results by this key — a retry after a network glitch or page refresh replays the original result rather than issuing a second charge. Disable idempotency (preset above), checkout, use the browser Back button to resubmit, and observe the double-charge.

Why idempotency is only on payment: the promotion and inventory services are read-only or explicitly compensated — retrying them is always safe without a key. Payment is the only call that both has a side effect (charges money) and may receive retries without a fresh user action.

In production: real gateways (Stripe, Adyen) accept an Idempotency-Key HTTP header. Generate it once, store it in your outbox alongside the order attempt, and replay on every retry.

5 — Intermittent Failures and Retry Safety

FLAKY mode fails every other call — simulating an unstable network partition or a host that is restarting. For read-only operations (promotion lookup) this is safe to retry immediately: the FLAKY + graceful-degrade combination means every other checkout silently skips the discount with no data corruption. For write operations (payment) you must check idempotency before retrying — otherwise a successful charge that returned a network error will be charged again.

In production: Resilience4j @Retry with exponential back-off and jitter handles transient failures automatically. Always annotate retried methods with whether they are idempotent — only retry non-idempotent operations (payment) if an idempotency key is in place.

Two Sources of Truth for Stock

Inventory stub (ConcurrentHashMap)
Authoritative for checkout. reserveStock decrements it immediately; releaseReservation restores it. Prevents overselling under concurrent sessions.

Product table (H2 DB)
Display value shown on product pages. Decremented only after a confirmed order via ProductRepository.decrementStock() — not at reservation time.

When the inventory commit fails and a full rollback occurs, the DB stock counter is not decremented — the saga's compensation steps restore consistency without requiring a distributed lock or two-phase commit.

What a Production System Would Add
  • Outbox table for compensations — write the compensating event to the DB in the same transaction as the order; a background worker retries it until it succeeds, surviving JVM crashes.
  • Timeouts on every external call — not just payment. Wrap inventory and promotion calls with a CompletableFuture or Resilience4j @TimeLimiter.
  • Circuit breakers — Resilience4j @CircuitBreaker opens after N consecutive failures, failing fast without blocking threads, and half-opens periodically to probe recovery.
  • Retry with back-off + jitter — Resilience4j @Retry on idempotent calls; never retry payment without an idempotency key in place.
  • Distributed tracing — propagate a trace ID (OpenTelemetry / Micrometer) across all service calls so a failed saga can be reconstructed end-to-end in Grafana or Jaeger.
  • Dead-letter queue — compensating actions that fail after N retries land in a DLQ for manual review rather than being silently dropped.
  • Idempotency key in the outbox — store the payment idempotency key alongside the pending order so every replay uses the same key.
