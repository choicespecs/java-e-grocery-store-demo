## Commands

```bash
# Run the application (starts on http://localhost:8080)
./mvnw spring-boot:run

# Build and package
./mvnw package

# Run all tests
./mvnw test

# Run a single test class
./mvnw test -Dtest=GroceryApplicationTests

# Skip tests during build
./mvnw package -DskipTests
```

The Maven wrapper (`mvnw`) is auto-generated by Spring Boot on first build. If it is missing, use `mvn` directly.

**H2 console** is available at `http://localhost:8080/h2-console` while the app is running.
JDBC URL: `jdbc:h2:mem:grocerydb`, username `sa`, no password.

## Architecture

Spring Boot 3 / Java 17 / Thymeleaf monolith demonstrating how a checkout flow interacts with external services and handles failures through compensating transactions (the Saga pattern).

### Layer map

```
controller/          HTTP → Thymeleaf rendering; catches domain exceptions and maps to UI messages
  CartModelAdvice    ControllerAdvice that injects cartItemCount + categories into every template
  DemoController     GET/POST /demo — reads and writes DemoFaultConfig; applies scenario presets
service/             Business logic; CheckoutService owns the checkout saga
external/{name}/     Interface + stub per external system (inventory, payment, promotions, cart deals)
  {name}/stub/       The only @Service implementing each interface; swap for real impl here
demo/                Fault injection: FaultMode (NORMAL|DOWN|SLOW|FLAKY), DemoFaultConfig, FaultInjector, SagaTrace
domain/              JPA entities: Category, Product, Order, OrderItem, OrderStatus
repository/          Spring Data JPA; ProductRepository has a custom @Modifying stock decrement
dto/                 Cart (@SessionScope Spring bean) + CartItem (session-serializable snapshot)
config/              DataInitializer seeds categories, products, and the inventory stub on startup
exception/           Typed exception per failure mode; caught selectively in CheckoutController
```

### The checkout saga — `CheckoutService`

The most important class in the codebase. It coordinates three external services in sequence with explicit compensation at each failure point:

```
1. inventoryService.checkAvailability()   ← read-only pre-flight, no side effects
2. promotionService.applyPromotion()      ← throws InvalidPromotionException on bad code
3. inventoryService.reserveStock()        ← SIDE EFFECT: reduces available stock
   └─ payment fails → releaseReservation  (COMPENSATION: stock returned to pool)
4. paymentService.processPayment()        ← SIDE EFFECT: charges the card
   └─ persist fails → refund + release    (COMPENSATION)
5. orderService.createOrder()             ← writes to DB (@Transactional)
   └─ commit fails → refund + cancel + release
6. inventoryService.commitReservation()   ← finalises the hold (removes the reservation record)
```

`CheckoutService` is intentionally **not** `@Transactional` — the saga spans external calls that cannot be wrapped in a single DB transaction. Each individual DB write carries its own `@Transactional`.

Key design fact: payment runs on a dedicated daemon-thread executor with a `Future.get(paymentTimeoutMs)` deadline. Inventory calls run directly on the servlet request thread with no timeout. This asymmetry is intentional demo material — see the *Inventory SLOW at reserve* preset.

### External service pattern

Each external service follows the same structure:

```
external/payment/
  PaymentService.java          ← interface
  PaymentRequest.java          ← request record
  PaymentResult.java           ← response record
  PaymentStatus.java           ← enum
  stub/StubPaymentService.java ← @Service implementation (in-memory simulation)
```

Spring autowires the stub automatically because it is the only `@Service` implementing the interface. To swap in a real implementation, annotate the new class `@Service @Profile("prod")` and annotate the stub `@Profile("!prod")`.

### Cart promotions (`CartPromotionService`)

Automatic quantity-based deals — no promo code required. Evaluated on every cart mutation (add / update / remove) inside `CartController`, which calls `cartPromotionService.evaluate(cart.getItems())` and stores the result back on the session-scoped `Cart`. The `Cart` exposes `getCartPromotionDiscount()` and `getEffectiveTotal()` for the templates and for `CheckoutService`, which subtracts the cart discount from the final charged amount.

### Chaos Engineering Panel (`/demo`)

`DemoFaultConfig` is a singleton `@Component` holding volatile fault settings for every stub. Stubs read it on each call; `DemoController` reads and writes it. `FaultInjector.apply(mode, label, slowMs, flakyCounter)` is the single dispatch point — it throws `ServiceUnavailableException` for DOWN/FLAKY, or sleeps for SLOW.

Fault modes: `NORMAL` | `DOWN` | `SLOW` | `FLAKY`

Inventory has three independent fault points (`checkAvailability`, `reserveStock`, `commitReservation`). `releaseReservation` is never fault-injected — compensations must be reliable.

One-click presets (handled in `DemoController.applyPreset`):
- `all-normal`, `payment-down`, `payment-slow`, `payment-timeout`, `payment-flaky`, `payment-no-idempotency`
- `inventory-check-down`, `inventory-check-flaky`, `inventory-check-slow`
- `inventory-reserve-down`, `inventory-reserve-flaky`, `inventory-reserve-slow`
- `inventory-commit-down`, `inventory-commit-slow`, `inventory-commit-flaky`
- `promotion-slow`, `promotion-down-graceful`, `promotion-down-hard`, `promotion-flaky`
- `coupon-down`, `coupon-flaky`, `coupon-slow`

`SagaTrace` accumulates a per-request step log (`ok`, `failed`, `compensated`, `skipped`, `degraded`). The controller adds it to the model on any checkout failure so the template renders a colour-coded execution trace.

### Cart (session state)

`Cart` is a `@SessionScope @Component`. Spring creates one instance per HTTP session via CGLIB proxy. It stores `CartItem` snapshots (productId, name, **price at time of adding**) so cart totals don't shift if a product's price changes mid-session.

Controllers inject `Cart` directly. Services receive it as a parameter — services never hold session state.

### Stock: two sources of truth

| Source | Role |
|--------|------|
| `Product.stockQuantity` (H2 DB) | Cached display value; updated by `ProductRepository.decrementStock()` after a confirmed order |
| `StubInventoryService.available` (ConcurrentHashMap) | Authoritative for checkout; seeded from DB by `DataInitializer` on startup |

The two-phase flow (reserve → commit/release) prevents overselling when concurrent sessions buy the same item.
