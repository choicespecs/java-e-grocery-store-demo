CartController

Controller

Handles all cart mutations: add, update quantity, remove, and clear. After every mutation it calls refreshDiscounts() which re-evaluates both CartPromotionService (quantity deals) and CouponService (coupon offers). The coupon engine is gracefully degraded — failures clear coupon discounts without aborting the operation.

RoutesGET/POST /cart, /cart/add, /cart/update, /cart/remove, /cart/clear
Session beanCart (@SessionScope)
Fault-tolerantCouponService (graceful degrade)
Discount Refresh
private void refreshDiscounts() {
cart.applyCartPromotion(
cartPromotionService.evaluate(items));
try {
cart.applyCoupons(couponService.evaluate(items));
} catch (ServiceUnavailableException e) {
cart.applyCoupons(CouponResult.none());
cart.setCouponServiceDown(true);
}
}

OrderService

Service

Owns all @Transactional DB writes related to orders: createOrder (persists Order + OrderItems, decrements Product.stockQuantity) and cancelOrder (sets status to CANCELLED). CheckoutService calls this after a successful payment but before committing the inventory reservation.

createOrder@Transactional — writes Order + OrderItems
cancelOrder@Transactional — sets CANCELLED status
decrementStockProductRepository.decrementStock() custom query

CheckoutService

Service — Saga Orchestrator

The most important class. Coordinates a 6-step saga across InventoryService, PromotionService, PaymentService, and OrderService. NOT @Transactional — the saga spans external calls that cannot be wrapped in a single DB transaction. Each step either succeeds and advances the saga, or triggers explicit compensating actions to undo already-committed side effects.

Saga Steps
Step 1checkAvailability — read-only, no side effects
Step 2applyPromotion — read-only, graceful degrade OK
Step 3reserveStock — SIDE EFFECT: reduces stock
Step 4processPayment — SIDE EFFECT: charges card (dedicated thread)
Step 5createOrder — DB write (@Transactional)
Step 6commitReservation — makes hold permanent

f.cancel(true);
inventoryService.releaseReservation(r);
throw new ServiceUnavailableException(...);
}

StubInventoryService

External Stub — Inventory

Simulates an external warehouse inventory system using two ConcurrentHashMaps: one for available stock, one for active reservations. Implements a two-phase reserve/commit protocol to prevent overselling under concurrent sessions. releaseReservation intentionally has no fault injection — compensating transactions must always succeed.

checkAvailabilityRead-only. Fault: CHECK mode
reserveStockDeducts available. Fault: RESERVE mode
commitReservationRemoves hold. Fault: COMMIT mode
releaseReservationReturns stock. NO fault injection
Data Structure
ConcurrentHashMap<Long, Integer> available;
// productId -> units not yet reserved

ConcurrentHashMap<String, Map<Long,Integer>> reservations;
// reservationId -> {productId -> qty}

In-Memory Stock (ConcurrentHashMap)

Persistence — In-Memory

The authoritative source of truth for stock availability during checkout. Held in StubInventoryService as a ConcurrentHashMap. Seeded from the H2 DB by DataInitializer on startup. Reserve/release operations happen here in real-time, enabling the two-phase reserve/commit flow that prevents overselling under concurrent sessions.

AuthorityCheckout availability checks
Synced with H2On startup; after confirmed orders
Thread-safeConcurrentHashMap + merge()
LifetimeJVM process (resets on restart)

HTTP Session (@SessionScope Cart)

Session State

Cart is a @SessionScope @Component — Spring creates one CGLIB-proxied instance per HTTP session. Stores CartItem snapshots (productId, name, price at time of adding) so cart totals don't shift if a product's price changes mid-session. Controllers inject Cart directly; services receive it as a method parameter and never hold it as state.

Scope@SessionScope (one per HTTP session)
ProxyCGLIB — injected as singleton-scoped proxy
Price snapshotPrice locked at time of cart.addItem()
DiscountscartPromotionDiscount + couponDiscount
